# Manual Versioning — Spec

## 1. Goal

Enable full version history for manuals so that every publish creates an immutable snapshot. Users can browse version history, view past versions read-only, compare two versions side-by-side with diff highlighting, and roll back to any previous version.

## 2. Dependencies

- **Epic 1** — Auth, roles, app shell
- **Epic 2** — Manual CRUD, Tiptap editor, publish flow, data model

## 3. Tech Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Version storage | Full JSON snapshot per version | Simple retrieval, no reconstruction needed, acceptable size for manual content |
| Diff engine | `diff-match-patch` (npm) | Word/sentence-level diffing, lightweight, well-maintained |
| Change summary | Auto-generated by diffing consecutive snapshots server-side | No user friction on publish; retroactive notes available |
| Version history UI | Side panel (Sheet) in editor | Contextual, no navigation away from editor |
| Rollback strategy | Create new version (N+1) with old snapshot content | Preserves full audit trail, always reversible |
| Mobile diff | Toggle view — single pane with switch | Avoids cramped side-by-side on small screens |

## 4. Implementation Tasks

### 4.1 Prisma Schema — ManualVersion Model

1. Add `ManualVersion` model to `packages/db/prisma/schema.prisma`:
   ```prisma
   model ManualVersion {
     id          String   @id @default(cuid())
     manualId    String
     manual      Manual   @relation(fields: [manualId], references: [id], onDelete: Cascade)
     version     Int
     content     Json     // Full snapshot: { productName, overview, instructions, warnings }
     authorId    String
     author      User     @relation(fields: [authorId], references: [id])
     note        String?
     changeSummary String?
     createdAt   DateTime @default(now())

     @@unique([manualId, version])
     @@index([manualId])
     @@index([authorId])
   }
   ```
2. Add `versions ManualVersion[]` relation to the `Manual` model.
3. Add `manualVersions ManualVersion[]` relation to the `User` model.
4. Run `pnpm db:migrate:dev --name add-manual-version`.

### 4.2 Version Service

1. Create `apps/web/src/lib/services/version-service.ts`:
   ```typescript
   export async function createVersion(manualId: string, authorId: string): Promise<ManualVersion>
   export async function getVersionHistory(manualId: string): Promise<ManualVersion[]>
   export async function getVersion(manualId: string, versionNumber: number): Promise<ManualVersion | null>
   export async function rollbackToVersion(manualId: string, versionNumber: number, authorId: string): Promise<ManualVersion>
   export async function updateVersionNote(versionId: string, note: string): Promise<ManualVersion>
   ```
2. `createVersion`:
   - Read current manual content fields (productName, overview, instructions, warnings).
   - Determine next version number: `SELECT MAX(version) FROM ManualVersion WHERE manualId = ?` + 1 (or 1 if none).
   - Serialize all content as a JSON snapshot.
   - Generate change summary by diffing against previous version's snapshot (if exists).
   - Insert `ManualVersion` row.
3. `rollbackToVersion`:
   - Fetch target version snapshot.
   - Update the `Manual` row with the snapshot content.
   - Call `createVersion` to record the rollback as a new version.
   - Set change summary to `"Rolled back to version {N}"`.

### 4.3 Change Summary Generator

1. Create `apps/web/src/lib/utils/diff-summary.ts`:
   ```typescript
   import DiffMatchPatch from 'diff-match-patch'

   export function generateChangeSummary(
     prev: ManualSnapshot,
     curr: ManualSnapshot
   ): string
   ```
2. Compare each section (productName, overview, instructions, warnings) between snapshots.
3. Return a human-readable summary, e.g., `"Updated product overview, added 1 instruction, removed 1 warning"`.

### 4.4 Integrate Versioning into Publish Flow

1. Modify the existing publish Server Action in `apps/web/src/app/(dashboard)/manuals/[id]/actions.ts`:
   - After updating the manual's `status` to `PUBLISHED`, call `createVersion(manualId, session.user.id)`.
   - Wrap in a transaction so version creation is atomic with publish.

### 4.5 Version History API Routes

1. Create `apps/web/src/app/api/manuals/[id]/versions/route.ts`:
   - `GET` — returns version history for a manual (reverse chronological). Auth: any user with access to the manual.
2. Create `apps/web/src/app/api/manuals/[id]/versions/[version]/route.ts`:
   - `GET` — returns a single version snapshot. Auth: any user with access to the manual.
3. Create `apps/web/src/app/api/manuals/[id]/versions/[version]/rollback/route.ts`:
   - `POST` — triggers rollback. Auth: any user with edit access (assigned editor or admin).
4. Create `apps/web/src/app/api/manuals/[id]/versions/[versionId]/note/route.ts`:
   - `PATCH` — updates version note. Auth: any user with edit access.

### 4.6 Version History Side Panel

1. Create `apps/web/src/components/manuals/version-history-panel.tsx`:
   - shadcn/ui `Sheet` component, opens from the right.
   - Triggered by a clock icon button (`Clock` from `lucide-react`) in the editor header bar.
   - Fetches version list from `GET /api/manuals/[id]/versions`.
   - Renders reverse-chronological list: version number, date (relative, e.g., "2 hours ago"), author name, auto-generated change summary, optional note.
   - Each version row has actions: "View", "Compare", "Rollback".
   - Inline editable note field (pencil icon, saves on blur via `PATCH .../note`).
2. Add the clock icon button to the existing editor header in `apps/web/src/app/(dashboard)/manuals/[id]/page.tsx`.

### 4.7 Read-Only Version View

1. Create `apps/web/src/components/manuals/version-viewer.tsx`:
   - Receives a `ManualSnapshot` (JSON content) as prop.
   - Renders all manual sections in read-only mode using the same layout as the editor but with all fields disabled/non-editable.
   - Header shows: "Version {N}" badge, date, author, "Rollback to this version" button.
2. Open in a dialog (`Dialog` from shadcn/ui) or full-page overlay from the version history panel.

### 4.8 Version Comparison View

1. Install `diff-match-patch`:
   ```bash
   cd apps/web && pnpm add diff-match-patch && pnpm add -D @types/diff-match-patch
   ```
2. Create `apps/web/src/components/manuals/version-diff.tsx`:
   - Props: `leftVersion: ManualSnapshot`, `rightVersion: ManualSnapshot`, `leftLabel: string`, `rightLabel: string`.
   - Side-by-side layout on desktop (>= 1024px): left pane (older), right pane (newer).
   - Word/sentence-level diff highlighting: additions in green background, deletions in red background.
   - Diff each section independently (productName, overview, each instruction, each warning).
3. Create `apps/web/src/components/manuals/version-diff-mobile.tsx`:
   - Single-pane toggle view for mobile (< 1024px).
   - Toggle switch at top: "Version {A}" | "Version {B}".
   - Changed sections highlighted with a colored left border.
4. Users select two versions from the history panel for comparison.

### 4.9 Rollback Confirmation Dialog

1. Create `apps/web/src/components/manuals/rollback-dialog.tsx`:
   - shadcn/ui `AlertDialog`.
   - Title: "Rollback to Version {N}?"
   - Description: "This will replace the current manual content with the content from Version {N}. A new version will be created with the current content before rolling back. This action is reversible."
   - Actions: "Cancel" (secondary), "Rollback" (destructive variant).
   - On confirm: `POST /api/manuals/[id]/versions/[version]/rollback`.
   - Show success toast on completion, refresh editor content and version history.

### 4.10 Responsive Adjustments

1. Version history panel: full-width sheet on mobile (< 1024px), 400px-wide sheet on desktop.
2. Version comparison: automatically switch between side-by-side (desktop) and toggle view (mobile) using `useMediaQuery` or Tailwind responsive classes.
3. Version viewer dialog: full-screen on mobile.

## 5. API Contracts

### GET /api/manuals/[id]/versions

**Auth:** Any user with access to the manual.

```typescript
// Response 200
type VersionHistoryResponse = {
  versions: {
    id: string
    version: number
    authorId: string
    authorName: string
    note: string | null
    changeSummary: string | null
    createdAt: string // ISO 8601
  }[]
}
```

### GET /api/manuals/[id]/versions/[version]

**Auth:** Any user with access to the manual.

```typescript
// Response 200
type VersionDetailResponse = {
  id: string
  version: number
  manualId: string
  content: ManualSnapshot
  authorId: string
  authorName: string
  note: string | null
  changeSummary: string | null
  createdAt: string
}

type ManualSnapshot = {
  productName: string
  overview: object // Tiptap JSON
  instructions: {
    id: string
    title: string
    body: object // Tiptap JSON
    order: number
  }[]
  warnings: {
    id: string
    title: string
    description: string
    severity: "DANGER" | "WARNING" | "CAUTION"
    isCustom: boolean
    libraryWarningId: string | null
  }[]
}

// Response 404 — version not found
```

### POST /api/manuals/[id]/versions/[version]/rollback

**Auth:** User with edit access (assigned editor or admin).

```typescript
// Response 200
type RollbackResponse = {
  newVersion: {
    id: string
    version: number
    changeSummary: string
    createdAt: string
  }
}

// Response 403 — no edit access
// Response 404 — version not found
```

### PATCH /api/manuals/[id]/versions/[versionId]/note

**Auth:** User with edit access.

```typescript
// Request
type UpdateNoteRequest = {
  note: string
}

// Response 200
type UpdateNoteResponse = {
  id: string
  note: string
}
```

## 8. TDD Approach

All tests should be written before or alongside implementation, following red-green-refactor:

1. **Unit tests first** (`diff-summary.test.ts`): Write failing tests for `generateChangeSummary` covering identical snapshots, single-section changes, and multi-section changes. Then implement `diff-summary.ts` to make them pass.
2. **Integration tests second** (`version-service.test.ts`): Write failing tests for `createVersion`, `rollbackToVersion`, `updateVersionNote`, and access control. Then implement the version service and API routes.
3. **E2E tests last** (`version-history.spec.ts`, `version-compare.spec.ts`, `version-rollback.spec.ts`): Write Playwright tests against the acceptance criteria. Then implement the UI components to make them pass.

Each implementation task in Section 4 should not be considered complete until its corresponding tests pass.

## 11. Out of Scope

- Automatic version pruning or archiving (all versions kept forever)
- Branching or forking versions
- Collaborative merge of concurrent edits
- Version-level access control (if you can see the manual, you can see all versions)
- Export of version history
- Diff of non-text content (e.g., images)
- Real-time collaborative editing conflict resolution
